\chapter{Evaluación}
\label{chap:evaluacion}

\section{Preguntas de Investigación}

La evaluación se organiza en torno a tres preguntas de investigación:

\begin{description}
  \item[RQ1:] \textit{¿En qué medida el sistema genera traducciones IBDD sintácticamente válidas a partir de escenarios BDD?}

  Esta pregunta evalúa la capacidad del sistema para producir traducciones que se ajusten a la gramática formal IBDD. Dado que la validación sintáctica es determinística (realizada por el parser Earley), la variabilidad en los resultados proviene exclusivamente de la naturaleza estocástica del LLM. Por ello, la respuesta a esta pregunta requiere múltiples ejecuciones del pipeline sobre el mismo dataset para cuantificar tanto la tasa de éxito promedio como su dispersión.

  \item[RQ2:] \textit{¿Cómo incide el idioma del prompt y del dataset en la calidad de la traducción?}

  El sistema soporta prompts en inglés y en español, y los escenarios BDD pueden estar redactados en cualquiera de los dos idiomas. Esta pregunta investiga si la combinación de idiomas entre el prompt y el dataset afecta la tasa de validez sintáctica de las traducciones. La respuesta permite determinar si el sistema es robusto ante variaciones lingüísticas o si existe una configuración de idioma preferible.

  \item[RQ3:] \textit{¿Cuál es la efectividad del ciclo de corrección iterativo?}

  El pipeline incluye un mecanismo de corrección que analiza los errores sintácticos de las traducciones fallidas y reintenta la traducción con retroalimentación del diagnóstico. Esta pregunta evalúa cuántas de las traducciones inicialmente fallidas se corrigen mediante este ciclo, cuántas iteraciones son necesarias para converger y si existen casos que resisten la corrección de forma persistente.
\end{description}

Este diseño permite separar el efecto del idioma del prompt del efecto del idioma del dataset, y detectar si la coincidencia entre ambos idiomas incide en los resultados.

\subsection{Partición del dataset (desarrollo y prueba)}

Para evitar sesgos por ajuste sobre el mismo conjunto utilizado en la evaluación final, se adopta una partición fija del dataset de 120 escenarios en dos subconjuntos:

\begin{itemize}
  \item \textbf{Desarrollo (dev):} 30 escenarios.
  \item \textbf{Prueba (test):} 90 escenarios.
\end{itemize}

La partición se definió de forma consistente con el historial de construcción del dataset: el subconjunto \textit{dev} coincide con la versión inicial de 30 escenarios que fue utilizada durante las etapas tempranas de ajuste y depuración, mientras que los 90 escenarios restantes conforman el conjunto \textit{test} reservado para la evaluación final. En términos de archivos, esto se materializa en \texttt{Dataset\_dev\_30.json} y \texttt{Dataset\_test\_90.json}, así como sus equivalentes en español.

Una vez fijada la configuración del sistema sobre el conjunto \textit{dev}, las corridas reportadas en la Sección~\ref{sec:resultados} se realizarán exclusivamente sobre el conjunto \textit{test}.

\subsection{Repeticiones y manejo de la estocasticidad}

Los LLMs son modelos estocásticos: ante la misma entrada y con temperatura mayor a cero, pueden producir salidas distintas en cada ejecución. Una única corrida del pipeline no permite distinguir si un resultado refleja el comportamiento típico del sistema o una fluctuación aleatoria. Para obtener estimaciones confiables de las métricas, cada configuración se ejecuta $N$ veces de forma independiente sobre el conjunto \textit{test}.

En la implementación experimental de este trabajo, $N$ se parametriza desde línea de comandos y se registra automáticamente junto con los resultados agregados. Para la evaluación final se adopta inicialmente $N=5$ corridas por configuración. Esta cantidad se considera un compromiso razonable entre obtener una estimación con variabilidad observable (evitando un número demasiado bajo de corridas, estadísticamente poco representativo) y mantener la ejecución dentro de los límites de tiempo y del alcance de este trabajo.

A partir de las $N$ ejecuciones de cada configuración se calcula la media y la desviación estándar de cada métrica, y se construyen intervalos de confianza al 95\,\% mediante la distribución $t$ de Student.

\subsection{Ciclo de corrección}

En todas las configuraciones, el pipeline se ejecuta con un número máximo de rondas de corrección configurable mediante el parámetro \texttt{-{}-max-rounds}. En cada ronda, los casos que fallaron la validación sintáctica son procesados por el agente de análisis de errores y retraducidos con retroalimentación del diagnóstico. El ciclo se detiene cuando todos los casos superan la validación o cuando se alcanza el límite de rondas.

El valor de \texttt{-{}-max-rounds} también se parametriza y queda registrado en los metadatos de cada ejecución para garantizar trazabilidad y reproducibilidad.

Este diseño permite evaluar la convergencia del ciclo de corrección: para cada ejecución se registra cuántos casos se corrigen en cada ronda, lo que permite construir una curva de convergencia promediada sobre las $N$ repeticiones.

\subsection{Protocolo de ejecución}

Cada ejecución del pipeline sigue el protocolo descrito a continuación:

\begin{enumerate}
  \item Se invoca el pipeline con la configuración correspondiente (prompt, dataset, modelo, número máximo de rondas).
  \item El sistema traduce cada escenario BDD de forma individual, registrando el tiempo de traducción por caso.
  \item Se valida sintácticamente cada traducción mediante el parser Earley.
  \item Se ejecuta el ciclo de corrección iterativo hasta convergencia o agotamiento del límite de rondas.
  \item Se guardan las traducciones finales, los resultados de validación y las métricas del pipeline en archivos JSON.
\end{enumerate}

Previo a estas corridas de evaluación final sobre \textit{test}, se realizan corridas de desarrollo sobre \textit{dev} con el único objetivo de depuración y ajuste del enfoque. Dichas corridas no se incluyen en el reporte cuantitativo final.

El pipeline registra automáticamente las métricas de cada ejecución en un archivo \texttt{*\_metrics.json} que incluye: el modelo y proveedor utilizados, los resultados de validación iniciales y finales, el detalle por ronda de corrección (casos fallidos, casos corregidos, tiempo) y el tiempo total del pipeline.

\section{Métricas}

La evaluación utiliza métricas que capturan aspectos complementarios del desempeño del sistema. Se distinguen tres categorías: métricas de validez sintáctica, métricas del ciclo de corrección y métricas operativas. El criterio compartido entre estas categorías es que todas pueden calcularse a partir de la salida del parser y los metadatos registrados por el pipeline, sin requerir anotación humana ni traducciones de referencia. Esto garantiza objetividad y reproducibilidad de los resultados, a costa de no capturar la dimensión semántica de las traducciones.

\subsection{Validez sintáctica}

La métrica principal es la \textit{tasa de validez sintáctica} (SVR, \textit{Syntactic Validity Rate}), definida como la proporción de escenarios cuya traducción IBDD es aceptada por el parser Earley:

\begin{equation}
\text{SVR} = \frac{\text{número de traducciones sintácticamente válidas}}{\text{número total de escenarios}}
\label{eq:svr}
\end{equation}

Se distinguen dos variantes de esta métrica. $\text{SVR}_\text{inicial}$ mide la tasa de validez tras la primera traducción, sin correcciones. $\text{SVR}_\text{final}$ mide la tasa de validez tras completar el ciclo de corrección iterativo. La diferencia $\text{SVR}_\text{final} - \text{SVR}_\text{inicial}$ cuantifica la ganancia atribuible al mecanismo de corrección.

Para cada configuración, se reportan la media y la desviación estándar de ambas variantes sobre las $N$ ejecuciones independientes.

La elección de SVR como métrica principal se apoya en tres razones. En primer lugar, el resultado del parser Earley es determinístico ante una traducción dada: no interviene ningún juicio subjetivo en la medición, lo que garantiza reproducibilidad independientemente del evaluador. En segundo lugar, es directamente relevante para el objetivo del sistema, cuya función es producir traducciones conformes a la gramática IBDD. En tercer lugar, no requiere un corpus de traducciones de referencia anotadas manualmente, lo que resulta relevante dado que no existe un repositorio público de escenarios BDD con traducciones IBDD validadas.

\subsection{Ciclo de corrección}

Para evaluar la efectividad del ciclo de corrección (RQ3) se definen las siguientes métricas:

\begin{description}
  \item[Tasa de recuperación:] proporción de traducciones inicialmente fallidas que se corrigen tras el ciclo completo. Formalmente:
  \begin{equation}
    \text{RR} = \frac{\text{fallidas iniciales} - \text{fallidas finales}}{\text{fallidas iniciales}}
    \label{eq:rr}
  \end{equation}

  \item[Rondas hasta convergencia:] número de rondas de corrección necesarias hasta que no se producen correcciones adicionales o se alcanza el límite configurado. Esta métrica indica la velocidad de convergencia del ciclo.

  \item[Correcciones por ronda:] cantidad de casos corregidos en cada ronda individual. Permite construir una curva de convergencia que muestra si la mayoría de las correcciones ocurren en las primeras rondas o si se distribuyen de forma uniforme.
\end{description}

La inclusión de estas tres métricas responde a la necesidad de evaluar el ciclo de corrección de manera cuantitativa y complementaria: RR sintetiza el beneficio neto del mecanismo, las rondas hasta convergencia caracterizan su velocidad, y la curva de correcciones por ronda revela si las mejoras se concentran en las primeras iteraciones o requieren múltiples rondas. Sin este conjunto de métricas no sería posible determinar si el ciclo aporta un beneficio real o si los escenarios que fallan en la primera traducción continúan fallando independientemente del número de intentos.

\subsection{Estabilidad por caso}

Para identificar escenarios que presentan dificultad persistente para el sistema, se analiza la estabilidad de los resultados a nivel de caso individual. Para cada escenario $i$ se calcula la proporción de ejecuciones (sobre las $N$ repeticiones) en las que su traducción resultó sintácticamente válida tras el ciclo completo:

\begin{equation}
  \text{Estabilidad}(i) = \frac{\text{ejecuciones en que el caso } i \text{ fue válido}}{N}
\label{eq:estabilidad}
\end{equation}

Un caso con estabilidad igual a 1 se traduce correctamente en todas las ejecuciones; un caso con estabilidad cercana a 0 resiste sistemáticamente la traducción. Esta métrica permite identificar patrones en los escenarios que resultan problemáticos para el LLM, como estructuras sintácticas inusuales o niveles de complejidad que exceden la capacidad del modelo. La estabilidad complementa la visión agregada de SVR con un análisis a nivel de caso individual, necesario para distinguir fallos aleatorios, que se resuelven en ejecuciones posteriores, de fallos sistemáticos, que persisten independientemente de la estocasticidad del modelo.

\subsection{Métricas operativas}

Se registra el tiempo de traducción por caso y el tiempo total del pipeline por ejecución. Si bien estas métricas no inciden directamente en la calidad de la traducción, permiten evaluar la viabilidad práctica del sistema y cuantificar la sobrecarga computacional introducida por el ciclo de corrección iterativo. En particular, la comparación del tiempo promedio por ronda permite estimar si el costo de cada iteración adicional se justifica por la ganancia en SVR que produce.

\subsection{Casos no convergentes tras tres rondas}

El análisis de los archivos de validación finales (\texttt{*\_validation.json}) permite identificar los casos que permanecen inválidos al agotar \texttt{max-rounds=3}. Sobre 20 ejecuciones totales (4 configuraciones $\times$ 5 corridas), se registran 220 instancias inválidas finales. No obstante, la no convergencia persistente es acotada: solo dos casos fallan en las cinco corridas de una misma configuración, el caso 68 en ES--EN y el caso 47 en ES--ES.

En estos casos persistentes, los mensajes de error se concentran en patrones sintácticos recurrentes: uso no esperado de corchetes en expresiones (\texttt{No terminal matches '['}), separación inválida de argumentos/listas (\texttt{No terminal matches ','}) y expresiones incompletas al final de la entrada (\texttt{Unexpected end-of-input}). En el conjunto completo de errores residuales, estos patrones aparecen con alta frecuencia (48, 33 y 34 ocurrencias, respectivamente), junto con fallos de sintaxis de gates (\texttt{?}/\texttt{!}, 12 ocurrencias).

Estos resultados muestran que, aunque el ciclo iterativo corrige una fracción importante de fallos iniciales, persiste un subconjunto de escenarios con estructuras que el sistema no estabiliza dentro de tres rondas. Esta evidencia delimita el principal frente de mejora del enfoque actual: robustecer la generación y/o las reglas de corrección para construcciones con listas, indexación y expresiones compuestas en guardas.

\section{Análisis}

El análisis se realiza exclusivamente sobre métricas agregadas de validación y error obtenidas en \textit{test} (90 casos).

\subsection{RQ1: validez sintáctica de las traducciones}

En las cuatro configuraciones, la validez sintáctica final media supera el 84\,\%: EN--ES alcanza 91.11\,\%, EN--EN 89.78\,\%, ES--EN 86.00\,\% y ES--ES 84.22\,\%. Bajo el protocolo definido ($N=5$, \texttt{max-rounds=3}), estos valores ubican la conformidad sintáctica final del sistema en el rango 84.22--91.11\,\% sobre el conjunto \textit{test}.

La dispersión de $\text{SVR}_{fin}$ es baja (DE entre 1.11 y 2.65 puntos), lo que sugiere un comportamiento relativamente estable entre corridas para la métrica principal. En términos de robustez, las configuraciones con prompt en inglés (EN--EN y EN--ES) muestran mejores resultados finales y baja variabilidad entre corridas.

\subsection{RQ2: efecto del idioma de prompt y dataset}

Los resultados muestran una diferencia clara en $\text{SVR}_{ini}$ según el idioma del prompt: las configuraciones con prompt en inglés (EN--EN, EN--ES) parten de niveles iniciales altos (78.00\,\% y 75.11\,\%), mientras que las configuraciones con prompt en español (ES--EN, ES--ES) parten de niveles iniciales significativamente menores (45.78\,\% y 46.00\,\%).

Tras el ciclo de corrección, la brecha se reduce en $\text{SVR}_{fin}$: las cuatro configuraciones quedan en el rango 84.22--91.11\,\%. Por lo tanto, en este experimento el idioma del prompt se asocia fuertemente con la calidad inicial, pero su efecto sobre la calidad final es menor una vez aplicado el mecanismo iterativo. El idioma del dataset también incide, aunque con efecto más moderado que el observado para el prompt en la etapa inicial.

\subsection{RQ3: efectividad del ciclo de corrección iterativo}

La ganancia media en puntos porcentuales confirma que el ciclo de corrección aporta mejoras en todas las configuraciones: +11.78 (EN--EN), +40.22 (ES--EN), +16.00 (EN--ES) y +38.22 (ES--ES). Asimismo, la tasa de recuperación de casos inicialmente fallidos es positiva en todas las configuraciones, con valores entre 53.03\,\% y 73.99\,\%.

No obstante, la métrica \texttt{rounds\_used.mean} es 3.00 en todas las configuraciones, lo que indica que en promedio se utiliza el máximo de rondas permitido. Esto sugiere que el ciclo mejora de forma sustantiva la validez, pero no alcanza convergencia temprana bajo el límite actual de iteraciones.

\subsection{Interpretación de resultados}

Los resultados sobre el conjunto \textit{test} (90 casos) muestran que el enfoque híbrido propuesto es efectivo para producir traducciones IBDD sintácticamente válidas en las cuatro configuraciones experimentales. En términos agregados, la validez final se ubicó entre 84.22\,\% y 91.11\,\% (Tabla~\ref{tab:resultados-calidad-test90}), lo que indica que la combinación de generación con LLM y validación formal permite sostener un nivel alto de conformidad con la gramática objetivo.

Un hallazgo central es la diferencia entre desempeño inicial y final. Las configuraciones con prompt en español parten de una validez inicial sustancialmente menor, pero recuperan gran parte de esa brecha tras aplicar corrección iterativa. En consecuencia, el mecanismo de reintento no actúa como un ajuste marginal, sino como un componente estructural del pipeline para estabilizar la calidad final entre configuraciones con condiciones lingüísticas distintas.

La interpretación de RQ2 sugiere que el idioma del prompt está más asociado con la calidad inicial que el idioma del dataset. No obstante, dado el tamaño muestral por configuración ($N=5$), esta evidencia debe leerse como patrón empírico consistente dentro del protocolo adoptado y no como una afirmación causal general sobre cualquier modelo, prompt o dominio.

Desde la perspectiva operativa, todas las configuraciones consumen en promedio el máximo de rondas permitido (\texttt{max-rounds=3}), lo que indica que el sistema mejora de manera sistemática, pero no converge tempranamente en el régimen actual. Este comportamiento es coherente con la existencia de un subconjunto de casos persistentemente difíciles, donde el diagnóstico y reintento corrigen parte de los errores, aunque no eliminan completamente la falla sintáctica residual.

En relación con trabajos previos, los resultados acompañan la literatura que reporta mejor desempeño de modelos GPT en tareas de generación estructurada para contextos BDD, especialmente cuando se utiliza prompting con ejemplos. Al mismo tiempo, este trabajo extiende ese marco al incorporar validación formal determinística y corrección iterativa en un pipeline end-to-end BDD$\rightarrow$IBDD, aspecto menos desarrollado en estudios centrados únicamente en calidad textual de salida.

\subsection{Limitaciones}

La principal limitación del estudio es que la evaluación cuantitativa se concentra en validez sintáctica. Una traducción aceptada por la gramática IBDD puede contener desalineaciones semánticas respecto del escenario BDD original, por lo que la métrica SVR no agota la noción de calidad de traducción.

Una segunda limitación es el tamaño del muestreo estocástico por configuración ($N=5$). Este diseño permite estimar media, dispersión e intervalos de confianza, pero restringe el poder para comparaciones inferenciales más finas entre configuraciones cercanas.

También debe considerarse la dependencia del prompt y del conjunto de reglas de traducción utilizados. Parte del desempeño observado refleja decisiones de diseño del prompt y del mecanismo de corrección, por lo que cambios en estas piezas pueden modificar los resultados sin alterar la arquitectura general.

Finalmente, la evaluación se realizó con un único modelo seleccionado y un conjunto acotado de dominios dentro del dataset construido. Aunque esto es adecuado para la pregunta de investigación de esta tesis, no garantiza transferencia automática a otros modelos, proveedores o colecciones de escenarios con distribuciones lingüísticas distintas.

\subsection{Consideraciones de validez}

Para la validez interna, se mitigaron riesgos mediante partición fija dev/test, repetición de corridas por configuración y uso de un validador sintáctico determinístico. Aun así, persisten posibles fuentes de variación asociadas a la naturaleza estocástica del modelo y a cambios no controlados en el entorno del proveedor.

Para la validez de constructo, la métrica principal (SVR) representa de forma directa la conformidad gramatical, pero no captura fidelidad semántica. Por lo tanto, las conclusiones deben entenderse estrictamente en términos de corrección sintáctica de la traducción.

Para la validez externa, los hallazgos se sustentan en un dataset de 120 escenarios con evaluación final sobre 90 casos de test y cuatro configuraciones lingüísticas. Este alcance ofrece evidencia relevante para el contexto estudiado, pero su generalización a otros dominios, tamaños de dataset o configuraciones de prompting requiere validación adicional.
